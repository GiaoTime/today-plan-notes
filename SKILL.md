# Today Plan Notes

把零散待办整理成一份可以直接执行的中文日计划，并优先适配 macOS「备忘录」的原生格式与核对清单样式。

## 什么时候使用

当用户出现以下意图时，使用这个 skill：

- 明确提到这些关键词之一：
  - 今日事项
  - 今日待办
  - 今日工作
  - 待办事项
  - 今日待办事项
  - 工作安排
  - 今日安排
  - 备忘录待办
  - Today's tasks
  - today's to-do list
  - today's work
  - to-do items
  - today's to-do items
  - work schedule
  - today's schedule
  - memo to-do
- 想整理当天待办
- 想把粗略主题扩展成可执行计划
- 想补充每项任务的预计耗时
- 想安排建议时间段或优先级
- 想把内容写进 macOS「备忘录」

## 目标

输出一份中文、简洁、可直接粘贴的日计划。默认要让用户打开备忘录后就能直接使用，不需要再二次改写。

## 默认指向 Apple Notes

除非用户明确说明只需要文本输出，否则当用户说“帮我整理今日待办”“整理今日待办”“今日待办”“今日事项”“今日工作”“待办事项”“今日待办事项”“工作安排”“今日安排”“备忘录待办”等短句时，默认含义是操作 macOS「备忘录」。

默认工作流：

- 优先读取 macOS「备忘录」中当天最新的相关笔记
- 相关笔记标题优先匹配：`今日待办事项`、`今日待办事件`、`今日事项`、`今日待办`、`今日工作`、`待办事项`、`工作安排`、`今日安排`
- 从笔记里提取现有待办主题
- 按本 skill 的标准补全预计耗时、建议时间段、工作流程、Codex可直接执行和今日优先级建议
- 默认写回同一条备忘录
- 如果当前环境无法直接控制或读取「备忘录」，要明确告诉用户无法执行桌面操作，并退而生成可粘贴版本
- 不要把“帮我整理今日待办”仅理解成普通文本生成任务

## 默认规则

- 输出语言固定为中文
- 输出默认适配 macOS「备忘录」原生格式
- 每个任务默认生成以下字段：
  - 序号
  - 任务名称
  - 预计耗时
  - 建议时间段
  - 工作流程
  - Codex可直接执行
- 默认补充一段“今日优先级建议”
- 如果用户没有给出具体时间边界，按标准工作日做合理安排
- 如果任务描述模糊，优先按常见办公类、内容类、工具配置类任务做合理拆解
- 如果当天任务明显过多，主动提醒用户压缩、延后或拆到次日

## 工作方式

### 1. 先识别任务类型

优先判断任务属于哪一类，再决定如何拆：

- 内容整理类：文案、方案、简历、页面内容、脚本、文章
- 工具使用类：安装、下载、配置、试跑、验证
- 资料整理类：筛选、分类、归档、记录来源
- 优化修改类：改版、优化、迭代、自查、收尾

### 2. 再补齐执行流程

工作流程优先写成 4 到 6 个连续动作，顺序要自然。默认优先使用以下动词：

- 明确
- 梳理
- 检查
- 试跑
- 修改
- 上传
- 验证
- 复查
- 记录
- 收尾

不要把流程写成空泛口号，要尽量让用户一看就能开工。

### 2.5 再补一个「Codex可直接执行」

每个任务默认再补 3 到 5 个可以直接交给 Codex 的动作，重点写那些能帮助用户更快完工、减少重复劳动、适合标准化执行的内容。

优先写这些类型：

- 文案整理
- 结构补全
- 文件检查
- 格式统一
- 信息归类
- 脚本生成
- 代码修改
- 备忘录整理
- 上传前检查
- 结果复查

写法要求：

- 只写 Codex 真的适合执行的动作
- 不要重复“工作流程”原句
- 不要写空泛话术，比如“提供帮助”“协助完成”
- 优先写可直接落地、可快速交付的动作
- 单条尽量控制在 3 到 5 个分号分隔动作里

### 3. 估算耗时

没有更多上下文时，使用以下经验值：

- 很轻的小任务：20-30 分钟
- 一般单点任务：30-45 分钟
- 中等复杂任务：45-60 分钟
- 需要思考和修改的任务：1-1.5 小时
- 需要完整产出和复查的任务：1.5-2.5 小时

如果一个任务看起来过大，优先拆小，不要硬塞进一个超长时间块。

进一步估时时，优先参考下面这些判断维度：

- 只需要执行，不需要判断：优先估 20-30 分钟
- 需要少量判断或筛选：优先估 30-45 分钟
- 需要试错、验证、配置：优先估 45-60 分钟
- 需要明显的内容产出或结构调整：优先估 1-1.5 小时
- 需要从整理、修改到复查的完整闭环：优先估 1.5-2.5 小时

也可以按任务类型快速估时：

- 下载、安装、收集、简单记录：20-30 分钟
- 筛选资料、试用工具、基础整理：30-45 分钟
- 工具配置、试跑、排查、初步优化：45-60 分钟
- 文案修改、页面优化、简历调整、结构梳理：1-1.5 小时
- 完整提交物、成品输出、上传交付、自查复核：1.5-2.5 小时

估时修正规则：

- 如果任务依赖外部资源、网络、下载速度、登录权限，增加 10-15 分钟缓冲
- 如果任务第一次做、流程不熟，增加一个档位
- 如果任务只是补充、微调、复查，降低一个档位
- 如果同类任务连续处理，后续任务可略微缩短
- 默认每半天保留 15-30 分钟机动时间，不要把时间排满

### 4. 安排建议时间段

默认遵循以下原则：

- 上午优先放高价值、需要专注的任务
- 下午前段适合处理中等复杂度任务
- 下午后段适合下载、配置、复查、收尾类任务
- 任务之间尽量保持自然衔接

如果用户已经给出明确时间边界，就以用户安排为准。

没有明确时间边界时，优先按下面这个工作日骨架分配：

- 09:30-11:00：高专注、高产出任务
- 11:00-12:00：中等强度任务或承接任务
- 14:00-15:30：需要判断、修改、试跑的任务
- 15:30-17:00：配置、整理、跟进、补充类任务
- 17:00-18:00：复查、收尾、上传、记录类任务

分配时间段时，优先遵守以下细则：

- 第一个时间块尽量给最重要的主任务
- 同类任务尽量放在相邻时间段，减少切换成本
- 需要网络、下载、安装的任务，尽量不要占用上午最黄金的深度工作时间
- 需要创作、思考、修改的任务，尽量不要放到最后一个时间块
- 当天如果有 1 个超重任务，只给它 1 个主时间块，再把剩余工作拆到其他时段或提醒顺延
- 当天如果有多个轻任务，可以把 2 个轻任务合并到同一时间段内表达

时间段生成规则：

- 20-30 分钟任务：优先放在整点后的短时间块里
- 30-45 分钟任务：可单独成段，也可与轻任务拼接
- 45-60 分钟任务：通常单独分配一个时间块
- 1-1.5 小时任务：优先放上午或下午第一段
- 1.5-2.5 小时任务：如果用户任务较多，优先拆成“主处理 + 复查收尾”两个阶段表达

如果用户当天任务总量已经明显超过正常工作日：

- 不要勉强给每项都塞进完整时间段
- 要明确指出哪些适合延后
- 优先保住前 3 个最重要任务的时间块完整性

### 5. 给出优先级建议

默认按以下标准排序：

- 先做产出价值高、必须推进的任务
- 再做支持性任务
- 最后做配置、下载、收尾类任务

优先级建议只写一句，直接告诉用户先做什么、再做什么、最后做什么。

## 输出格式

默认使用下面这个结构，直接输出中文正文，不要加解释性前言：

- 笔记总标题“今日待办事项”使用备忘录“标题”样式
- 日期单独一行，可作为补充说明行保留
- 每个任务标题前加两位序号，格式固定为 `01 `、`02 `、`03 `
- 每个任务标题统一使用备忘录“标题”样式
- “预计耗时 / 建议时间段 / 工作流程”统一使用备忘录“项目符号列表”
- “Codex可直接执行”统一单独一行，使用备忘录“等宽样式”
- 不再默认追求原生圆框核对清单，优先保证轻量、高效、稳定的版式输出

```text
今日待办事项

2026-04-23

01 任务名称
- 预计耗时：xx
- 建议时间段：xx
- 工作流程：xx；xx；xx；xx。
Codex可直接执行：xx；xx；xx。

02 任务名称
- 预计耗时：xx
- 建议时间段：xx
- 工作流程：xx；xx；xx；xx。
Codex可直接执行：xx；xx；xx。

今日优先级建议：xx。
```

## 模糊任务处理

如果用户只给了非常粗的主题：

- 先按最常见的完成路径补全
- 不要因为缺少细节就停住
- 用保守但可执行的方式估时
- 必要时在工作流程里体现“先确认”“先梳理”“先检查”

## 任务过多时的处理

如果总任务量明显超过一天合理负荷：

- 先照常整理
- 然后补一句提醒，建议压缩到 3 个最重要任务，或把低优先级任务顺延

## 写入备忘录时的约定

如果要直接操作 macOS「备忘录」：

- 优先走脚本化写入，而不是纯手工 UI 点击
- 默认使用 `scripts/render_today_plan_note.py`
- 输入只需要整理后的任务数据：任务名称、预计耗时、建议时间段、工作流程、今日优先级建议
- 如果任务数据来自实时整理结果，默认补出“Codex可直接执行”
- 序号默认按两位数递增生成，从 `01` 开始
- 脚本先写入“标题 + 日期 + 带序号的任务标题 + 项目符号列表 + 等宽执行块”的基础版式
- “预计耗时 / 建议时间段 / 工作流程”统一按项目符号列表写入
- “Codex可直接执行 / 今日优先级建议”统一写成“等宽样式”
- 如果当前会话可用 `Computer Use`，脚本写入后立即补最后一层自动化：
  - 逐个选中主任务标题行
  - 执行备忘录原生“制作核对清单”动作
  - 不改动下方等宽说明块
- 原生圆框补充默认设置 1 分钟上限：
  - 如果 1 分钟内看起来无法完成全部主任务的核对清单转换，立即停止
  - 不为了圆框效果牺牲整体效率
  - 这时保留“主任务标题 + 等宽说明块”的轻量版结果即可
- 如果当前会话不可用 `Computer Use`，保留脚本输出的主任务标题和等宽说明块，并明确告知用户“原生圆框核对清单”这一步待下次可控桌面时再补
- 避免逐段手工点选格式，除非用户明确要求即时人工微调

## 风格要求

- 中文自然、明确、可执行
- 不要写得像项目管理术语堆砌
- 不要输出过度冗长的解释
- 让用户一眼就知道今天怎么开始做
